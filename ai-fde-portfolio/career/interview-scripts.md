# 面试表达素材

## 1. 这个项目和普通 API 中转有什么区别？

普通 API 中转解决的是“能不能调到模型”，但企业真实上线后更关心“能不能稳定、便宜、可追踪、可审计、可持续优化”。

所以我把项目定位成 LLM Gateway / LLMOps Mini Platform，核心能力不是转发，而是治理：

- 路由：按成本、速度、健康状态选择模型。
- 稳定性：失败后 fallback，记录每一次尝试。
- 成本：记录 Token 和费用，按应用统计。
- 商业化：预算控制、超预算降级。
- 质量：Prompt 版本和 Eval Harness。

这类能力能支撑客户套餐、毛利控制、故障排查和模型效果优化。

## 2. 为什么需要模型路由？

企业客户的请求并不都是同一个价值等级。有些任务需要强模型，有些任务只需要低价模型。如果所有请求都走高价模型，成本不可控；如果所有请求都走低价模型，质量和稳定性又可能不够。

模型路由的价值在于：

- 低价值任务走低价模型，控制成本。
- 高价值任务走强模型，保证效果。
- 某个供应商异常时切换备用模型，提升可用性。
- 后续可以结合模型健康度、失败率、客户套餐做更细的策略。

## 3. 如何做成本控制？

我会分三层做：

- 记录层：每次调用记录 app、model、prompt_tokens、completion_tokens、cost。
- 策略层：按应用设置月度预算、QPS、默认模型和降级模型。
- 产品层：将成本和客户套餐绑定，比如基础套餐默认低价模型，高级套餐开放更强模型和更高并发。

这样成本控制不是单纯技术问题，而是和平台商业化、客户分层、毛利管理连接起来。

## 4. 为什么 Prompt 版本重要？

企业项目里 Prompt 经常会变，如果没有版本，线上效果变差后很难排查。Prompt 版本可以做到：

- 每次调用关联具体版本。
- 出问题时能定位是模型变了、数据变了，还是 Prompt 变了。
- 配合评测集做回归测试。
- 支持灰度和回滚。

这也是从 Demo 到生产可用必须补的一环。

## 5. 如何看待最近比较火的 Harness？

我理解的 harness 不是一个新名词，也不是单纯测试脚本，而是模型外面的工程和产品系统。企业 AI 项目里，模型只是推理能力，harness 决定这个能力如何被测试、约束、追踪、回归和持续改进。

我会分三层看：

- LLM Gateway 解决调用治理：路由、成本、稳定性、预算、Prompt 版本。
- Eval Harness 解决可重复评测：测试集、批量运行、通过率、成本、耗时、fallback、失败归因和发布门禁。
- RAG Evaluation Harness 解决知识库质量闭环：标准问答集、召回命中、引用可信度、幻觉风险、Bad Case 和文档版本回归。
- Agent Harness 解决执行环境：工具注册、状态管理、权限、轨迹、人工确认和失败恢复。

很多 Agent 失败并不是换更强模型就能解决，而是工具定义不清、状态没有保存、终止条件缺失、权限边界模糊，或者没有 trace 导致无法排查。

## 6. Harness 和普通测试脚本有什么区别？

普通测试脚本通常只验证某个函数或接口是否返回预期结果。Eval Harness 更关注 AI 系统的整体环境是否稳定可复现：

- 同一批 case 可以重复运行，做版本回归。
- 每个 case 记录模型、Prompt 版本、路由策略、attempts 和成本。
- 不只判断 pass/fail，还要做失败归因。
- 能比较不同模型、Prompt、路由、RAG 策略的效果差异。

这对 AI 项目很重要，因为模型输出不完全稳定，只看一次结果没有意义，必须看批量指标和失败模式。

## 7. 如果面试官问技术深度

可以这样答：

这个项目现在是标准库核心逻辑 + FastAPI API 层。核心服务里我把模型配置、应用配置、Prompt 版本、路由策略、provider 调用、日志存储和 Eval Harness 拆开。SQLite 用来记录调用日志，后续如果生产化，会换成 MySQL/PostgreSQL，加 Redis 做限流和缓存，用 Prometheus/Grafana 做监控，模型 provider 也会接真实 API。

我作为 AI 平台产品 / AI Evals & Harness 方向候选人，不会把重点放在炫技，而是会讲清楚为什么这些模块存在，以及它们分别解决企业 AI 应用从 Demo 到生产可用里的什么问题。

## 8. 为什么不把 FDE 作为主攻方向？

FDE 是一个真实趋势，但我不会把它作为主攻 title。原因是这个岗位的含金量非常依赖公司和产品成熟度：在顶级 AI 公司里，FDE 可能是深度嵌入客户、重构工作流和推动生产落地；但在本地中小公司里，也可能退化成驻场实施、客户成功、售前支持、交付救火和外包开发的混合岗位。

所以我会把 FDE 当成能力标签，而不是职业主线。我真正主攻的是 AI Evals / Harness / LLMOps 质量治理产品：利用我的开发、现场交付和产品经验，把客户现场的 bad case、生产 trace、模型失败和成本问题转成评测集、回归测试、失败归因、发布门禁和平台治理能力。

## 9. 你是否要转 AI 全栈？

我不会把自己定位成纯 AI 全栈工程师。AI 产品岗位确实越来越需要懂技术，甚至要能用 AI 辅助开发做原型、跑测试、理解代码约束，但“AI 全栈”在招聘语境里多数还是工程师 title。

我更准确的定位是 AI Builder 型 AI 平台产品经理：

- 产品上能判断用户、场景、流程、指标和商业价值。
- 技术上能理解 API、日志、路由、RAG、Agent、测试和 eval。
- 工具上能用 Codex / Claude Code / Cursor 把需求推进到可运行原型。
- 质量上能用测试、Eval Harness、trace 和失败归因证明这个原型不是玩具 Demo。

这样既保留我的产品差异化，也能回应市场对 AI 产品经理技术能力变强的要求。

## 10. LLM Gateway 项目 3 分钟讲法

我做这个项目的出发点是：企业接入大模型后，问题很快会从“能不能调用”变成“能不能稳定、可控、可追踪、可评测”。所以我把它设计成 LLM Gateway / LLMOps Mini Platform，而不是普通 API 中转。

核心能力有五块：

- 统一模型调用入口，屏蔽多个供应商和模型差异。
- 路由和 fallback，根据成本、速度、可用性选择模型，失败后切备用模型。
- 调用日志和成本统计，记录 app、model、provider、token、cost、latency、status。
- 预算、限流和 Prompt 版本，把平台治理和商业化、回滚、排障连起来。
- Eval Harness V0，用测试集批量运行，输出 pass rate、成本、耗时、fallback、trace 和失败归因。

我觉得这个项目最能体现我想做的方向：AI 平台产品不是只接模型，而是把模型能力变成可评测、可追踪、可回归、可持续优化的平台能力。

## 11. LLM Gateway 项目 8 分钟讲法

背景上，很多 AI 应用早期只封装一个 API，但企业上线后会遇到一批治理问题：不同模型成本差异很大，供应商稳定性不一致，Prompt 变更后效果难追踪，客户用量和成本难核算，模型效果也不能只靠人工看几条输出。

所以我把项目拆成三个层次：

第一层是调用治理。这里包括模型配置、应用配置、统一调用入口、路由策略和 fallback。路由不是技术炫技，而是产品策略：低价值任务可以走低价模型，高价值任务走更强模型，供应商异常时自动 fallback。

第二层是商业和运营治理。每次调用都会记录应用、模型、provider、token、cost、latency、status 和 prompt version。这样后续能做客户用量分析、预算控制、套餐设计、毛利分析和问题排查。

第三层是质量治理，也就是 Eval Harness V0。它会批量跑 case，输出通过率、失败率、平均成本、平均耗时、fallback 次数、trace 和 failure breakdown。V0 的 rubric 先从关键词、模型/供应商匹配、工具出现和预期失败标签开始；baseline 记录稳定版本指标；regression 用来发现 Prompt、模型或路由更新后的退化；CI gate 用来控制上线前质量门禁。

这个项目里，我个人最看重两个产品判断：

- 第一，模型能力不是产品能力，只有加上路由、日志、成本、权限、评测和回归，才是平台能力。
- 第二，Eval Harness 不是普通测试脚本，它要评估模型外部环境，包括 Prompt 版本、路由策略、fallback、成本、trace 和失败归因。

如果继续做，我会优先补三件事：模型健康度、可配置 CI gate、以及 RAG/Agent Harness 的生产 trace 回流。

## 12. LLM Gateway 项目 15 分钟讲法

15 分钟讲法可以按“背景 -> 用户 -> 架构 -> 关键设计 -> 指标 -> 取舍 -> 后续”的顺序展开。

背景：

企业做大模型平台，第一阶段通常只追求接入模型，但一旦进入多应用、多客户、多模型、多供应商阶段，就会遇到成本、稳定性、审计、效果追踪和版本回归问题。我的项目就是把普通 API 中转升级为 LLM Gateway / LLMOps Mini Platform。

用户：

- 平台产品/运营需要配置模型、应用、预算、路由和用量报表。
- 研发/运维需要排查失败、延迟和供应商异常。
- 企业客户关心稳定性、成本可控和效果可解释。
- AI Evals / Harness 产品需要把 bad case、trace 和反馈沉淀成评测集。

架构：

入口是 Gateway API，后面依次经过 App Policy、Prompt Version、Router、Provider Adapter、Call Logs 和 Eval Harness。当前核心逻辑已经能跑标准库测试，API 层使用 FastAPI，日志用 SQLite 做本地证明。

关键设计：

- 路由策略：default、low_cost、fastest、balanced，分别对应成本、速度、综合体验等目标。
- fallback：供应商失败后按应用配置切备用模型，记录 attempts。
- 成本统计：按 token 和模型单价估算成本，支撑套餐和毛利判断。
- 预算和限流：把平台治理和商业化连接起来。
- Prompt 版本：用于定位效果变化和支持回滚。
- Eval Harness：批量 case、trace、failure tag、summary 指标。

指标：

- 质量：pass rate、fail rate、quality_failure。
- 成本：avg_cost_usd、应用总成本、模型维度成本。
- 速度：avg_latency_ms。
- 稳定性：fallback_count、model_failure。
- 策略：routing_failure、policy_failure、prompt_failure。

取舍：

如果只追求质量，可以默认强模型，但成本可能失控；如果只追求低价，客户体验可能变差；如果只追求稳定，会增加 fallback 和多供应商复杂度。因此平台需要把成本、质量、稳定性做成可配置策略，而不是写死。

我个人贡献和能力表达：

这个项目体现的是 AI Builder 型产品能力：我能把产品需求拆成可运行原型、测试和 eval，再把技术结果翻译成产品价值。它不是为了证明我想转纯工程师，而是证明我能和工程深入协作，知道 AI 平台从 Demo 到生产可用需要哪些治理能力。

后续：

下一步我会把它延伸到 RAG Evaluation Harness 和 Agent Harness。RAG 侧关注标准问答集、引用可信度、幻觉风险和文档版本回归；Agent 侧关注 tool registry、状态、权限、trace、human review 和失败恢复。
